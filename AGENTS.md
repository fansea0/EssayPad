# AGENTS.md - AI 协作风格约定

> 这是给 AI 协作者的工作指南。读完后请按以下原则工作,有冲突以本文为准。

## 1. 沟通

### 角色

- 用户是**后端开发**。后端(Go / SQL / 接口 / 数据模型)可以详细说,客户端(SwiftUI / 视图树 / 动画细节)只讲「做了什么、改了哪、UI 长什么样」,不解释 SwiftUI 怎么实现。

### 风格

- **中文为主**。代码标识符/库名/英文术语保持英文,叙述和注释用中文。
- **简洁优先**。能用一句话说清的不写一段;能直接给方案不罗列所有选项。
- **给出推荐**。建议/方案/权衡后必须有一个「我的推荐」,不要把球踢回。
- **不废话**。不写「这是一个很好的问题」之类的客套;失败/不确定时直说。

### 展示方式

- **页面/UI 加功能时,优先用图**(ASCII 布局 / 树形结构 / 流程图)。文字描述"上方一个按钮左边一个图标"远不如一张图清楚。
- 后端流程、数据模型、API 调用链同理 —— 能用图不用文字。
- 图不需要精美,清楚就行;复杂场景用 mermaid / dot 画正式图。

## 2. 工作流

### 默认 Plan 优先

- 涉及多文件 / 架构影响 / 不归路决策(选库、选存储、换 API) → 必须先 `EnterPlanMode`,写 plan,等 `ExitPlanMode` 批准再动。
- 1-2 行的 typo / 注释 / 显然 bug / 字段重命名 → 直接做,不进 plan。
- 不确定边界时,主动 `AskUserQuestion` 问清楚,不要猜。

### 决策粒度

- **必须 confirm**(影响不可逆或产品方向): 存储后端 / 鉴权方案 / 通信协议 / 拆 / 不拆服务 / UI 主流程重写。
- **自主决定**(用户不在意细节): 变量命名 / 配色 / 局部布局 / 错误提示措辞 / 内部函数拆分。
- **二选一时**: 给出推荐 + 1 句理由,不要列 5 个选项让用户自己选。

### 后端关键内容 → 找用户排版

- 用户是后端开发,**API 设计、SQL schema、核心数据结构、错误码**这些是用户的专业领域,先给方案让用户审,不要拍板:
  - 新增 / 修改 API → 先 show 路径 + 请求/响应 JSON 形状 + 错误码,等用户确认
  - 新增 / 修改表结构 → 先 show 建表 SQL 或迁移 SQL,等用户确认
  - 命名(字段名 / 表名 / handler 名)→ 不擅自起,先给候选让用户挑
- 内部实现(handler / service / DAO 内部)可以自主决定。

### 对话结束 commit

- **每次对话结束**(用户消息触发完成节点),本次有代码改动就自动提交一个 commit。
- commit message 用中文,格式 `type(scope): 说明`(遵循项目已有风格),不需要 co-author。
- **多个独立改动合并成一个 commit**(本次对话的逻辑单元),不要每个文件一个。
- **半成品 / WIP 不 commit**;只有 build 通过、验证过才算完成。
- 用户中途手动 `git commit` 了 → 不再补 commit。

## 3. 产品视角

- **不被用户为难**。如果一个功能给用户增加 3 步操作才能完成主任务,先简化流程,再考虑功能完整性。
  - 例:Keychain 在 sandbox 下反复弹密码 → 改 UserDefaults 明文存,体验 > 安全洁癖(本地工具)。
- **重视 UX**。「能不能好看一点」「UI 要美观」是硬要求,不能糊弄。
- **不引入不必要的依赖**。每个新依赖都问"这个项目值得吗";能 30 行手写的不引第三方库。
- **离线优先 / 本地优先**。网络是 AI 周报的可选项,不是必需项。
- **可恢复 > 不可恢复**。删除 / 覆盖 / 关闭操作前想清楚,不能误操作;不可逆操作要 confirm。

## 4. 代码风格

### 优先复用现有模式

- 改之前先 `Read` 相关文件,看现有代码怎么写;不发明新模式。
- 已有 helper / 工具函数(比如 `KeychainManager` / `APIClient`)就直接用,不要重新造。
- 复制一段已有代码(比如 NSPanel 配置)优于写"更优雅的版本"。

### 注释

- 函数级注释写「做什么」,不写「怎么做的每一步」。
- 关键决策点 / 已知坑要写「为什么」,不是「做了什么」。
- 不写 `// FIXME` 却不修;看到就修或建任务。

### 命名

- 类/类型名 `PascalCase`,函数/变量 `camelCase`,常量用有意义的全名。
- 中文 UI 文本可以直接中文(本来就是给中文用户),但标识符英文。

## 5. 验证

- 写完代码必须 build 通才能说"完成"。
- 改后端 → `go build` + 实际 curl 一遍关键接口。
- 改客户端 → `swiftc` 编译过 + 启动 app 看主流程。
- 改完一处,立刻想"会不会影响别的地方",跑一下相关 flow。
- 不在 PR 描述里写"测试过了"如果没真测。

## 6. 沟通反馈

- 用户**改了 AI 的代码**(linter / user 手动改) → 不要回滚,除非用户明确要求。
- 用户**反复报同一个 bug**(比如菜单没出现报 3 次) → 这是信号,说明根因没找到,先暂停,重新诊断,不要继续试错。
- 用户**情绪化或简短指令**("别弹了,改明文") → 立即响应,不要为"更好的方案"辩解。
- 用户**问"为什么"** → 解释根因,不要只说"做了什么"。

## 7. 任务管理

- 复杂任务(>3 步)用 `TaskCreate` 建任务清单,边做边 `TaskUpdate`。
- 任务状态: `pending` → `in_progress` → `completed`,不要在 `in_progress` 留死。
- 任务标题是动作 + 对象(「修复:打开设置闪退」),不是抽象描述。
- 任务多的时候 `TaskList` 看一下避免重复。

## 8. 探索与外部信息

- 优先用本地 `Read` / `Grep` / `Bash`,不要为了简单问题 `WebSearch`。
- 真正需要查文档/版本兼容/API 用法时,再 `WebFetch` 精确查。
- 不要编造 API / 函数签名;不确定就 `Read` 文件确认。

## 9. 不做的事

- ❌ 不在 plan / commit message 里写「首先...其次...最后...」这种 AI 八股。
- ❌ 不写「可能的方案: A / B / C」让用户选然后就停;必须推荐一个。
- ❌ 不为"完整性"加用户没要的功能(YAGNI)。
- ❌ 不在用户没要求时改 `TaskRow` / `TasksView` 等已有 UI。
- ❌ 不为了通过 CI 加无意义的测试;测试要为验证行为而存在。

## 10. 当前项目特定约定 (EssayPad)

- **mac 端是主力**,后端最小化 —— API 协议以 `docs/interface.md` 为准,改前先看。
- **数据本地优先**,SQLite 单文件,后端跑 127.0.0.1;不上云,不持久化到外部服务。
- **AI 配置存 UserDefaults 明文**(本地工具,不值得为 API key 弹密码)。
- **任务时长统计后端 SQL 实时算** `SUM(actual_minutes)`,不要在 Go 内存里维护副本。
- **统计用进程内内存**(重启清零)就够了,不要为此加 DB 表。
- **菜单快捷键**:`⌘⇧P` 倒计时 / `⌘⇧U` 正计时 / `⌘⇧R` 5min 休息;新增前先看 `EssayPadApp.swift` 是否有冲突。
- **不引入新依赖**(没有 SPM/Podfile/Cartfile),所有第三方能力都用手写或现有 Go 库。
